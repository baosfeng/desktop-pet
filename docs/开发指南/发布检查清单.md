# 发布检查清单

> 每次发布新版本前，逐项确认以下内容。
> 适用于版本号 ≥ 0.1.0 的所有发布。

---

## 发布前检查

### 代码质量

- [ ] `make lint` 全部通过（Go + Rust + TS）
- [ ] `make test` 全部通过（含 `-race`）
- [ ] 覆盖率不低于 60%
- [ ] CI（`.github/workflows/ci.yml`）最近一次运行全部绿色

### 版本与 Changelog

- [ ] `CHANGELOG.md` 已更新，包含本次所有变更
- [ ] 版本号遵循 semver 规范（当前 0.y.z）
- [ ] git tag 已创建：`git tag v0.x.y`

### 构建验证

- [ ] `make build-petcore-all` 编译通过（所有平台）
- [ ] `make build-frontend` 构建通过
- [ ] `cargo tauri build --bundles dmg` 打包通过

### 安全

- [ ] `cargo audit` 无高危漏洞
- [ ] Dependabot PR 已合并或已确认可忽略
- [ ] API Key 未硬编码在代码中（`grep -r "sk-" src/`）

### 文档

- [ ] `README.md` 中的版本号和下载链接已更新
- [ ] 新功能有关联的设计文档（在 `docs/superpowers/specs/` 中）
- [ ] 过时的设计文档已清理

---

## 发布步骤

```bash
# 1. 确认在 main 分支且工作区干净
git checkout main
git status

# 2. 更新 CHANGELOG.md
#    确保 Unreleased 下的条目已移到 v0.x.y

# 3. 打标签
git tag -a v0.1.0 -m "v0.1.0"
git push origin v0.1.0
#    → 自动触发 release.yml，构建并上传到 GitHub Releases

# 4. 验证发布
#    - GitHub Releases 页面确认产物已上传
#    - 下载并安装测试
#    - 确认自动更新可检测到新版本

# 5. 发布后
#    - 合并 release PR 到 develop
#    - 在项目公告中通知用户
```

## 回滚步骤

```bash
# 如果发布后发现严重问题：
# 1. 在 GitHub Releases 中标记该版本为 "Pre-release"
# 2. 重新发布上一个稳定版本（打新 patch 标签）
# 3. 已更新的用户将自动检测到降级
```
