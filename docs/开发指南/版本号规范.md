# 版本号规范

> 文档状态：已生效
> 生效日期：2026-07-11
> 适用范围：desktop-pet 全项目（Go PetCore / Tauri Rust 壳 / 前端）

## 版本格式

```
MAJOR.MINOR.PATCH
  │      │      └── PATCH: Bug 修复、微小调整（向后兼容）
  │      └───────── MINOR: 新功能、小改动（向后兼容）
  └──────────────── MAJOR: 大改动、不兼容变更
```

## 升级规则

| 变更类型 | 升级位 | 示例 |
|---------|--------|------|
| 💥 **不兼容 API 变更** | MAJOR +1 | 1.2.3 → **2.0.0** |
| ✨ **新功能**（向后兼容） | MINOR +1 | 1.2.3 → **1.3.0** |
| 🐛 **Bug 修复** | PATCH +1 | 1.2.3 → **1.2.4** |
| 🔧 **配置/CI 变更** | PATCH +1 | 1.2.3 → **1.2.4** |
| 📝 **文档更新** | 不升级 | 仅提交文档 |
| ♻️ **重构**（无行为变化） | PATCH +1 | 1.2.3 → **1.2.4** |

## 初始阶段（0.y.z）

在 `MAJOR = 0` 的初始开发阶段：

- **MINOR** 升级代表新功能/大改动（相当于正式版的 major）
- **PATCH** 升级代表修复/微小调整
- API 不稳定，任何版本都可能包含不兼容变更

## 版本文件

项目根目录的 `VERSION` 文件是版本号的**单一数据源**（Single Source of Truth）：

```
VERSION          → 0.0.0    # 纯文本，仅版本号
```

## 同步的目标文件

执行 `./scripts/bump-version.sh` 会自动同步以下文件：

| 文件 | 字段 | 说明 |
|------|------|------|
| `VERSION` | 文件内容 | 单一数据源 |
| `src-tauri/Cargo.toml` | `version` | Rust 包版本 |
| `src-tauri/tauri.conf.json` | `version` | Tauri 应用版本（显示在「关于」中） |

> ⚠️ **不要手动修改**以上文件的版本号，始终使用 `bump-version.sh` 脚本。

## 使用方式

```bash
# 查看当前版本
./scripts/bump-version.sh

# 升级补丁版本（bug 修复）
./scripts/bump-version.sh patch

# 升级次版本（新功能）
./scripts/bump-version.sh minor

# 升级主版本（不兼容变更）
./scripts/bump-version.sh major

# 设置指定版本
./scripts/bump-version.sh set 1.0.0
```

## 版本与 Git Tag

每次创建 Release 时，Git tag 使用 `v` 前缀：

```
v0.1.0    ← 对应版本 0.1.0
v1.0.0    ← 对应版本 1.0.0
```

Tag 创建由 CI 自动完成（`auto-release.yml`），或手动执行：

```bash
git tag v$(cat VERSION)
git push origin v$(cat VERSION)
```

## 发布流程

1. 确认所有 PR 已合并到 `main`
2. 运行 `./scripts/bump-version.sh <major|minor|patch>` 升级版本
3. 提交并推送版本变更
4. CI 自动触发打包、编译并发版
   - 详见 [自动发布工作流](../.github/workflows/auto-release.yml)
