# 事故响应流程

> 定义了项目中各类事故的严重级别、响应 SLA 和处理流程。
> 适用于桌面端应用（单机运行，非服务端）。

---

## 严重级别定义

| 级别 | 标签 | 定义 | 示例 |
|------|------|------|------|
| **SEV1** | 🔴 崩溃 | 应用完全不可用，或用户数据面临丢失风险 | 启动闪退、记忆数据损坏、对话完全不可用 |
| **SEV2** | 🟡 功能异常 | 核心功能异常，但有替代方案或临时绕过方式 | LLM 连不上（可降级到 Mock）、记忆不保存、Live2D 不显示 |
| **SEV3** | 🔵 小问题 | 非核心功能异常，不影响主要使用流程 | UI 错位、打字机效果卡顿、设置项不生效 |

## 响应 SLA

| 级别 | 响应时间 | 修复时间 | 更新频率 |
|------|---------|---------|---------|
| SEV1 | ≤ 30 分钟 | ≤ 4 小时 | 每 1 小时同步进展 |
| SEV2 | ≤ 4 小时 | ≤ 24 小时 | 每天同步 |
| SEV3 | ≤ 48 小时 | 下个版本 | 每个版本同步 |

> 响应时间 = 从事故报告到有人确认接手处理的时间。
> 修复时间 = 从接手到补丁发布（含 hotfix 发布）的时间。

## 事故处理流程

```
发现事故
  │
  ├── 判断严重级别（SEV1/SEV2/SEV3）
  │
  ├── 创建 GitHub Issue，添加对应 label（sev1/sev2/sev3）
  │
  ├── SEV1：立即停止当前工作，切换修复
  │   SEV2：当天内安排修复
  │   SEV3：纳入下个迭代计划
  │
  ├── 修复 → 提交 hotfix 分支 → CI 通过 → 发布补丁版本
  │
  └── SEV1/SEV2 修复后 48 小时内完成事后复盘
```

## Hotfix 发布流程

```bash
# 1. 从 main 切出 hotfix 分支
git checkout main
git checkout -b hotfix/xxx-简短描述

# 2. 修复代码 + 测试
# 3. 提交并推送
git commit -m "fix: 修复 xxx 问题"
git push origin hotfix/xxx

# 4. 创建 PR → Review → 合并到 main
# 5. 从 main 打补丁标签
git tag v0.1.1
git push origin v0.1.1  # 触发 Release workflow

# 6. 同时合并回 develop
git checkout develop
git merge main
```

## 升级路径

```
SEV1 处理人无法解决
    │
    └── → 联系项目负责人 (GitHub @maintainers)
              │
              └── → 仍无法解决 → 暂停功能开发，全团队投入
```

## 沟通

- 所有事故必须有一个公开的 GitHub Issue 跟踪
- SEV1/SEV2 的修复 PR 标签应添加 `hotfix`
- 用户可见的事故应在 Release Notes 中说明
