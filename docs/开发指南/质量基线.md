# 代码质量基线

> 本文档定义了项目各 Phase 必须满足的代码质量要求，贯穿所有功能开发。
> 这些要求不单独规划 Phase，而是在对应 Phase 编写代码时同步落地。

---

## Phase 1 要求（代码编写时同步落地）

| 要求 | 具体内容 | 验证方式 |
|------|---------|---------|
| **覆盖率 ≥60%** | Go + Rust + TS 三语言总体覆盖率不低于 60% | CI coverage job 自动检查 |
| **Lint 全部通过** | 三层 linter 不得有任何 error，且 **warning 数必须为 0**（`eslint . --max-warnings 0`） | CI lint job + pre-commit |
| **模糊测试 (Fuzz)** | 对 FSM 状态机、记忆系统核心数据结构编写 fuzz test | `go test -fuzz` 定期运行 |
| **错误追踪 (Sentry)** | Rust 壳和前端集成 Sentry SDK，捕获未处理异常 | opt-in 用户授权后上报 |
| **表格驱动测试** | Go 测试使用 table-driven tests 模式 | Code Review |
| **竞态检测通过** | Go 测试必须通过 `-race` 检测 | CI `go test -race` |

## Phase 2 要求

| 要求 | 具体内容 | 验证方式 |
|------|---------|---------|
| **e2e 集成测试** | 使用 Playwright 模拟用户操作关键路径 | CI e2e job |
| **基准回归检查** | 核心路径（对话、记忆检索）性能不退化 | Go `benchstat` / Rust `criterion` |
| **性能预算** | 打包体积不超 40MB，运行时内存不超 80MB | CI 构建后检查 |
| **依赖审计** | 零已知高危依赖 | Dependabot + cargo-audit 自动检测 |

## Phase 3+ 要求

| 要求 | 具体内容 | 验证方式 |
|------|---------|---------|
| **国际化 (i18n)** | 所有用户可见文本通过 `react-intl` 或 `i18next` 外部化 | Code Review |
| **无障碍 (a11y)** | 键盘导航完整、屏幕阅读器兼容 | `eslint-plugin-jsx-a11y` + 手动测试 |
| **安全审计** | 桌面控制操作可审计、可撤销 | Code Review + 权限模型检查 |

## 任务验收标准规范

> 所有任务（无论由 AI Agent 还是人工执行）在定义时必须附带可验证的验收标准。
> 验收标准必须在任务开始前写入设计文档或 Issue，作为完成的唯一评判依据。

### 验收标准必须包含的维度

| 维度 | 说明 | 示例 |
|------|------|------|
| **编译通过** | 对应语言构建工具通过 | `go build ./...`、`cargo build`、`pnpm build` |
| **Lint 通过** | 对应语言 linter 无 error | `golangci-lint run`、`cargo clippy`、`pnpm lint` |
| **测试通过** | 新增代码有对应测试，现有测试不破坏 | `go test -race ./...` |
| **测试覆盖率** | 新增代码覆盖率 ≥60%（增量覆盖） | `go test -coverprofile` |
| **边界情况** | 至少覆盖空输入、错误输入、并发场景 | table-driven tests 中的边界 case |
| **Mock 独立** | 测试不依赖外部服务（LLM/网络/文件系统） | 使用 `mock` Provider / `InMemoryManager` |

### 设计文档中的验收标准格式

每份设计文档末尾必须包含以下格式的验收标准：

```markdown
## 验收标准

- [ ] `go build ./...` 通过
- [ ] `golangci-lint run ./...` 无 error
- [ ] `go test -race ./...` 全部通过
- [ ] 新增代码覆盖率 ≥60%
- [ ] 边界情况已覆盖（空输入/错误输入/并发）
- [ ] 测试使用 Mock 而非真实外部服务
```

### 子 Agent 任务交付检查清单

子 Agent 完成任务后，必须提供以下证据：

1. 变更文件清单（paths）
2. 编译/测试命令的执行输出（verification）
3. 覆盖率满足要求的证明
4. 边界情况测试用例列表
