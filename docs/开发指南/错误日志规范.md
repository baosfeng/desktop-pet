# 错误日志规范

> **所有错误必须落日志，不得静默吞掉。**
>
> 凡是出现 `let _ =`、`_ , _ =`、`_ =` 吞掉可能错误的代码，
> 必须确认该表达式**绝不会产生错误**（如往 `Vec::new()` 写数据），
> 否则必须记录日志。

## 基本原则

| 原则 | 说明 |
|------|------|
| **所有 IO 错误必须记录** | 文件读写、网络请求、子进程通信、事件推送 |
| **所有序列化/反序列化错误必须记录** | JSON 解析、消息编解码 |
| **所有 sidecar 子进程通信错误必须记录** | stdin 写入失败、stdout 读取异常、subprocess 异常退出 |
| **所有 LLM Provider 调用错误必须记录** | API 请求失败、流式传输中断、token 超限 |
| **所有状态转换错误必须记录** | FSM 非法转换、插件启停失败 |
| **配置加载/热更新错误必须记录** | 配置格式错误、Provider 初始化失败 |

## Rust 规范

### ❌ 禁止的写法

```rust
// 静默忽略 IO 错误
let _ = child.write(data);
let _ = handle_clone.emit("pet:event", &event);
let _ = writer.send(&cmd);

// 静默忽略带上下文的错误
Err(_e) => { error!("generic message"); }
```

### ✅ 正确的写法

```rust
// IO 错误必须记录
if let Err(e) = child.write(data) {
    error!("sidecar write error: {e}");
}

// 事件推送错误必须记录（包含事件名称）
if let Err(e) = handle_clone.emit("pet:event", &event) {
    error!("failed to emit pet:event: {e}");
}

// writer 不可用必须记录（包含被丢弃的命令或消息）
if let Some(writer) = app_handle.try_state::<SidecarWriter>() {
    writer.send(&cmd);
} else {
    error!("chat message dropped: SidecarWriter not registered");
}

// 命令/子进程错误必须包含上下文信息
Err(e) => {
    error!("failed to spawn sidecar (attempt {}/{}): {e}", attempt + 1, max_retries);
}
```

### 必须包含的上下文

| 场景 | 必须包含的信息 |
|------|---------------|
| 子进程操作 | 二进制名称、尝试次数、具体错误 |
| 事件推送 | 事件名称（如 `pet:event`）、错误码 |
| sidecar 写入 | 消息摘要（前 100 字符）、具体错误 |
| JSON 解析 | 原始数据片段（避免敏感信息）、具体错误 |

## Go 规范

### ❌ 禁止的写法

```go
// 静默忽略所有可能的错误
_, _ = s.writer.Write(data)
_ = pCtx.Sink.Send(event)
_, _ = s.writer.Write(data)
```

### ✅ 正确的写法

```go
// IO 错误必须记录
if _, err := s.writer.Write(data); err != nil {
    s.log.Error("failed to write to stdout", "error", err)
}

// 事件推送错误必须记录
if err := pCtx.Sink.Send(event.Event{...}); err != nil {
    slog.Default().Error("pipeline: send thinking event failed", "error", err)
}
```

### 日志级别选择

| 级别 | 适用场景 |
|------|---------|
| `error!` / `log.Error` | 影响功能的错误：IO 失败、provider 调用失败、配置加载失败 |
| `warn!` / `log.Warn` | 非致命异常：非法的状态转换、fallback 到默认值、sidecar 重试 |
| `info!` / `log.Info` | 正常启动/关闭：sidecar 启动成功、配置热更新成功 |
| `debug!` / `log.Debug` | 诊断用详细日志 |

## 监控与排查

> 每次遇到「无响应」类问题，第一步检查日志：

```bash
# Rust/Tauri 日志（stderr）
cargo tauri dev 2>&1 | grep -E 'ERROR|error|warn'

# Go PetCore 日志（sidecar stderr）
# sidecar 的 stderr 会出现在 Tauri 前端控制台或终端
# 查看 Rust 端转发的 petcore stderr 日志
```

## 设计理由

- **用户无响应是最糟糕的体验** — 相比于错误的回复，完全没有反馈会让用户完全不知道发生了什么
- **日志是事后排查的唯一手段** — 很多错误在发生时无法（或不适合）直接展示给用户，必须记录到日志
- **错误上下文决定调试效率** — "write error" 没有意义，"sidecar stdin write error (attempt 2/3): broken pipe" 才有意义
