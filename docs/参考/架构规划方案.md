## 1. 总体架构

### 核心设计理念：**内核与壳分离**

参考 Reasonix（Go 内核 + CLI/Desktop/Server 三壳）和 Glances（GlancesCore + 6 种输出模式）的设计，整个系统分为两层：

```
                    ┌──────────────────────────────────────────┐
                    │              PetCore (内核)                │
                    │                                           │
                    │  ┌─────────┐  ┌──────────┐  ┌─────────┐  │
                    │  │ 状态引擎  │  │ AI Agent │  │ 记忆系统 │  │
                    │  │ (FSM)   │  │ (会话管理)│  │ (3层)   │  │
                    │  └─────────┘  └──────────┘  └─────────┘  │
                    │  ┌─────────┐  ┌──────────┐  ┌─────────┐  │
                    │  │ 插件系统  │  │ 工具系统  │  │ 事件总线 │  │
                    │  │ (MCP)   │  │ (Registry)│  │ (Event) │  │
                    │  └─────────┘  └──────────┘  └─────────┘  │
                    └────────────────────┬──────────────────────┘
                                         │
                         事件流 (event.Sink) + 命令 API
                                         │
              ┌──────────────────────────┼──────────────────────────┐
              │                          │                          │
              ▼                          ▼                          ▼
    ┌──────────────────┐    ┌────────────────────┐    ┌──────────────────┐
    │ Desktop Shell     │    │ CLI Shell          │    │ Server Shell     │
    │ (Wails/Tauri/     │    │ (Terminal TUI)     │    │ (Headless API)   │
    │  Electron)        │    │ 调试/管理/快捷操作  │    │ 远程访问/集成    │
    │ 可视化桌宠         │    │                    │    │                  │
    └──────────────────┘    └────────────────────┘    └──────────────────┘
```

### 关键设计原则

| 原则 | 说明 | 参考项目 |
|------|------|---------|
| **内核无 UI 依赖** | PetCore 不 import 任何 UI 库，只依赖 `event.Sink` 接口 | Reasonix `controller.go` |
| **壳是薄层** | Shell 只做初始化 + 事件消费，不包含业务逻辑 | Glances `standalone.py` (208行) |
| **配置驱动** | 所有行为通过配置文件声明，不硬编码 | Reasonix TOML, Espanso YAML |
| **事件驱动** | 内核发出事件，各 Shell 按需消费 | Reasonix Event, Glances 多输出 |
| **热加载优先** | 插件/动作/配置变更无需重启 | Espanso `notify` 文件监视 |

```

├── 窗口管理器          ← 透明窗口、穿透、置顶、拖拽、多屏

└── 系统托盘            ← 常驻、唤出、退出
```

**可参考的壳实现方案：**

| 方案 | 优点 | 缺点 | 参考项目 |
|------|------|------|---------|
| **Tauri v2** | 体积小(~15MB)、内存低(<50MB)、鼠标穿透原生支持、自动更新内置 | 需学 Rust | bongo-cat-next |
| **Wails v2** | Go 语言、React 前端、Reasonix 验证 | 鼠标穿透需自研、自动更新需自研 | Reasonix |
| **Electron** | 生态最成熟、API 齐全 | 体积大(~150MB)、内存高(~100MB+) | git2968/Desktop-pet |

### 3.2 CLI Shell（辅助壳）

CLI Shell 用于开发调试和快速操作，不要求图形界面：

```

子命令：
  pet chat              ← 交互式对话（REPL）
  pet run "..."         ← 单次对话
  pet status            ← 查看宠物状态
  pet memory query "..."← 查询记忆
  pet memory list       ← 列出记忆
  pet config set k=v    ← 修改配置
  pet plugin list       ← 列出插件
```

**技术方案：** Go 的 bubbletea（Reasonix 所用）或 Python 的 prompt_toolkit

### 3.3 Server Shell（扩展壳）

提供 HTTP/WebSocket API，支持远程访问或作为其他应用的 AI 后端：

```
API 端点：
  POST /api/chat         ← 对话接口
  GET  /api/status       ← 宠物状态
  WS   /ws               ← WebSocket 实时通信
  GET  /api/memory       ← 记忆查询
  POST /api/memory       ← 记忆修改
```

**技术方案：** Go net/http + gorilla/websocket 或 Python FastAPI

## 4. 插件系统

### 4.1 三层架构

参考 Espanso（YAML 包 + 热重载）、CopyQ（JS 脚本引擎）、Reasonix（MCP 子进程）的设计，结合桌宠场景设计三层插件体系：

```
┌────────────────────────────────────────────────────────┐
│  L1: 声明式动作包 (YAML)                                │
│  描述：配置驱动的动作/表情/行为组合，无需代码              │
│  格式：YAML 文件，定义触发器 + 动作序列                   │
│  热加载：文件监控自动重载                                 │
│  示例：pet_actions/hello.yaml                            │
│  适用：非开发者用户                                     │
├────────────────────────────────────────────────────────┤
│  L2: 脚本插件 (JS/Lua)                                  │
│  描述：轻量级脚本扩展，可编程但安全                       │
│  格式：单文件 JS 脚本，暴露简单 API                        │
│  热加载：文件修改后自动重载                               │
│  沙箱：限制文件/网络访问                                  │
│  适用：进阶用户                                         │
├────────────────────────────────────────────────────────┤
│  L3: MCP 工具/外部进程                                   │
│  描述：标准 MCP 协议，子进程隔离运行                       │
│  协议：stdio JSON-RPC                                   │
│  沙箱：独立进程 + 权限声明                                │
│  适用：开发者、生态扩展                                   │
│  参考：Reasonix MCP 客户端、5ire                          │
└────────────────────────────────────────────────────────┘
```

### 4.2 插件发现机制

```
启动时：
  1. 扫描 actions/ 目录 → 加载 *.yaml → L1 插件
  2. 扫描 scripts/ 目录 → 加载 *.js  → L2 插件
  3. 读取 config.toml → 启动 MCP 子进程 → L3 插件

运行时：
  - fsnotify 监控 actions/ 和 scripts/ 目录
  - 文件变更 → 自动重载（无需重启）
  - MCP 子进程崩溃 → 自动重启
```

## 5. LLM 适配层

### 5.1 Provider 接口

参考 Reasonix 和 NyaDeskPet 的设计，统一 Provider 接口：

```go
type Provider interface {
    // 基本信息
    Name() string
    Model() string

    // 流式对话（核心方法）
    Stream(ctx context.Context, req Request) (<-chan Chunk, error)

    // 非流式对话（备选）
    Chat(ctx context.Context, req Request) (Response, error)
}

type Request struct {
    Messages    []Message
    SystemPrompt string
    Tools       []Tool
    Temperature float64
    MaxTokens   int
}

type Chunk struct {
    Type    ChunkType    // Text / Reasoning / ToolCall / Usage / Done / Error
    Text    string
    ToolCall *ToolCall
    Usage   *Usage
}

type Message struct {
    Role    string   // system / user / assistant / tool
    Content string
    Images  []Image  // 可选：多模态
}
```

### 5.2 注册表模式

```go
// 全局注册表，各 Provider 在 init() 中自注册
var registry = make(map[string]ProviderFactory)

func Register(kind string, factory ProviderFactory) {
    registry[kind] = factory
}

func NewProvider(kind string, cfg ProviderConfig) (Provider, error) {
    factory, ok := registry[kind]
    if !ok {
        return nil, fmt.Errorf("unknown provider: %s", kind)
    }
    return factory(cfg)
}

// 使用方式（在 main.go 中 blank import）：
import _ "internal/llm/openai"   // openai.init() 调用 Register("openai", ...)
import _ "internal/llm/ollama"   // ollama.init() 调用 Register("ollama", ...)
```

### 5.3 Agent Pipeline（洋葱模型）

参考 NyaDeskPet 的 Pipeline 和 Open-LLM-VTuber 的装饰器链：

```
用户输入
   │
   ▼
┌─────────────────────┐
│  PreProcess Stage    │ ← 消息标准化、敏感词过滤、日志
├─────────────────────┤
│  Memory Stage        │ ← 注入相关记忆（短期 + 长期检索）
├─────────────────────┤
│  Context Stage       │ ← 构建 system prompt + 工具定义
├─────────────────────┤
│  LLM Call Stage      │ ← 调用 Provider.Stream()
│  (流式循环)          │
│  ┌───────────────┐  │
│  │ Text Chunk     │──├──► 实时显示（打字机效果）
│  │ Tool Call     │──├──► 执行工具 → 结果注入 → 继续
│  │ Done           │──├──► 完成
│  └───────────────┘  │
├─────────────────────┤
│  PostProcess Stage   │ ← 提取新事实 → 写入记忆
├─────────────────────┤
│  Response Stage      │ ← 触发宠物动作/情绪反应
└─────────────────────┘
   │
   ▼
输出给 Shell
```

### 5.4 工具调用循环

```go
func (a *Agent) Run(ctx context.Context, req Request) error {
    // 主循环
    for turn := 0; turn < maxToolTurns; turn++ {
        stream, err := a.provider.Stream(ctx, req)
        if err != nil { return err }

        var toolCalls []ToolCall
        for chunk := range stream {
            switch chunk.Type {
            case ChunkText:
                a.emit(EventAgentReply{text: chunk.Text})
            case ChunkToolCall:
                toolCalls = append(toolCalls, *chunk.ToolCall)
            case ChunkDone:
                // 本轮完成
            }
        }

        if len(toolCalls) == 0 {
            break // 没有工具调用，对话结束
        }

        // 执行工具
        for _, tc := range toolCalls {
            result := a.toolRegistry.Execute(ctx, tc)
            req.Messages = append(req.Messages,
                Message{Role: "assistant", ToolCalls: []ToolCall{tc}},
                Message{Role: "tool", ToolCallID: tc.ID, Content: result},
            )
        }
    }
}
```

## 6. 记忆系统

### 6.1 三层架构

参考设计规范 + MemOS + Reasonix + my-neuro 的综合设计：

```
┌──────────────────────────────────────────────────────┐
│  L1: 短期记忆（Short-Term）                           │
│  存储：内存（滑动窗口，最近 N 轮对话）                  │
│  生命期：当前会话                                      │
│  容量：最近 20 轮                                      │
│  检索：直接追加到上下文                                  │
├──────────────────────────────────────────────────────┤
│  L2: 核心记忆（Core）                                  │
│  存储：SQLite (结构化 key-value)                        │
│  生命期：永久                                          │
│  内容：用户姓名、偏好、生日、重要事实                      │
│  检索：启动时全量加载 + 增量更新                          │
│  修正：通过对话或 UI 直接修改                             │
├──────────────────────────────────────────────────────┤
│  L3: 长期记忆（Long-Term）                              │
│  存储：SQLite + 本地向量库                               │
│  生命期：长期（可衰减/归档）                              │
│  内容：对话摘要、历史事件、学到的习惯                       │
│  检索：语义相似度 + 关键词 BM25 混合检索                   │
│  策略：重要性评分 + 时间衰减 + 自动摘要                     │
└──────────────────────────────────────────────────────┘
```

### 6.2 记忆工作流

```
用户说话
   │
   ▼
L1 短期 ← 直接追加到上下文
   │
   ├── AI 在回复中发现新信息
   │      │
   │      ├── 结构化信息（姓名/偏好）→ L2 核心（SQLite）
   │      │
   │      └── 事件性信息 → 生成摘要碎片
   │             │
   │             ▼
   │          L3 长期 ← 向量化存储 + 时间戳
   │
   └── 下次对话
          │
          ▼
      1. 读取 L2 核心 → 用户偏好
      2. 检索 L3 长期 → 语义相似匹配
      3. 融合 → 构建 context → LLM 推理
```

### 6.3 事实提取策略

参考 MemOS 的提取机制和 Reasonix 的 `remember` 工具：

```go
// 方法1：LLM 主动提取（推荐）
// 每轮对话结束后，LLM 调用 remember 工具写入新发现的事实
tools: [
  {
    name: "remember",
    description: "记住一个关于用户的事实",
    parameters: {
      type: "string",  // 事实内容
      category: "preference|fact|event",  // 事实类别
      importance: 1-5  // 重要性
    }
  }
]
```

### 8.2 分离架构：WebSocket

如果保持 Electron + Python 后端分离，使用 WebSocket JSON 协议：

```json
// 前端 → 后端
{ "type": "user.message", "data": { "text": "你好" } }

{ "type": "pet.emotion", "data": { "mood": "happy", "intensity": 0.8 } }
```

### 8.3 统一事件格式

参考 Reasonix 的 `eventwire` 设计，无论用哪种 IPC，事件格式保持一致：

```go
type WireEvent struct {
    Kind string          `json:"kind"`       // 事件类型
    Data json.RawMessage `json:"data"`       // 事件数据
    Meta EventMeta       `json:"meta"`       // 元信息（时间戳、来源）
}

type EventMeta struct {
    Timestamp int64  `json:"ts"`
    Source    string `json:"source"`  // core / plugin / user
    SessionID string `json:"session_id"`
}
```

## 9. 技术方案对比与选型

### 9.1 全栈方案对比

| 方案 | 壳框架 | 后端语言 | 安装包 | 内存 | 鼠标穿透 | 自动更新 | Live2D | 开发效率 | 迁移成本 |
|------|--------|---------|--------|------|---------|---------|--------|---------|---------|
| **A: Tauri + Rust 后端** 🥇 | Tauri v2 | Rust | **~15MB** | **<50MB** | ✅原生 | ✅内置 | ✅WebView | 中 | 高 |
| **B: Wails + Go 后端** | Wails v2 | Go | **~20MB** | **~50MB** | ❌自研 | ❌自研 | ✅WebView | 高 | 中 |
| **C: Electron + Go 后端** | Electron | Go (子进程) | ~150MB | ~80-150MB | ✅ | ✅ | ✅PixiJS | 高 | 低 |
| **D: Electron + Python 后端** (当前) | Electron | Python | ~200MB+ | ~100-200MB+ | ✅ | ✅ | ✅PixiJS | **最高** | **零** |

### 9.2 推荐路径

```
Phase 1（当前）        Phase 2（可选）          Phase 3（长期）
┌─────────────┐      ┌──────────────┐       ┌──────────────────┐
│ Electron +  │ ──►  │ 优化打包体积  │ ──►   │ 考虑迁移至 Tauri │
│ Python 后端 │      │ 抽离 PetCore │       │ 或 Wails + Go   │
│ (快速迭代)  │      │ (Go 重写内核)│       │ (纯静态二进制)   │
└─────────────┘      └──────────────┘       └──────────────────┘
```

**Phase 1（当前）：** 保持 Electron + Python，快速实现 MVP
**Phase 2（内核抽取）：** 用 Go 重写 PetCore（状态机 + AI Agent + 记忆），作为子进程运行
**Phase 3（壳替换）：** 将 Electron 壳替换为 Tauri 或 Wails，实现 ~150MB → ~15MB 的体积缩减
