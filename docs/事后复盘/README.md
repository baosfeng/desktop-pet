# 事后复盘流程 (Blameless Postmortem)

> 每次 SEV1/SEV2 事故修复后 48 小时内，由处理人撰写事后复盘。
> 核心原则：**不追责，关注系统改进。** 所有复盘公开可查。

---

## 复盘模板

```markdown
# 事后复盘：事故标题

- **日期**: YYYY-MM-DD
- **严重级别**: SEV1 / SEV2
- **处理人**: @xxx
- **持续时间**: 从发现到修复共 XX 小时

## 概要

用 2-3 句话描述发生了什么。

## 时间线

| 时间 | 事件 |
|------|------|
| HH:MM | 发现事故 |
| HH:MM | 确认严重级别 |
| HH:MM | 定位根因 |
| HH:MM | 修复提交 |
| HH:MM | 补丁发布 |

## 根因分析

事故发生的直接原因是什么？

## 影响范围

- 受影响用户数/时长
- 是否有数据丢失

## 改进措施

| 措施 | 负责人 | 截止日期 |
|------|--------|---------|
| xxx | @xxx | YYYY-MM-DD |

## 经验教训

从这个事故中学到了什么？如何防止同类问题再次发生？
```

## 复盘原则

1. **不追责** — 问题出在流程/系统/代码，不是人
2. **公开透明** — 复盘文档放在 `docs/事后复盘/YYYY-MM-DD-标题.md`
3. **关注系统改进** — 每个事故必须产出一项系统层面的改进措施
4. **跟踪闭环** — 改进措施必须指定负责人和截止日期，记入 GitHub Issues

## 复盘文件归档

```
docs/事后复盘/
├── README.md         ← 本文件（复盘规范）
├── 2026-07-11-xxx.md  ← 复盘实例
└── ...
```
